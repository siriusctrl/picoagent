# ADR 0011: Separate Model Stream Idleness From the Request Deadline

- Status: Accepted
- Date: 2026-07-20
- Refines: ADR 0010 (model request failure boundaries)

## Context

Picoagent originally wrapped every complete provider future in one 300-second
timeout. The deadline included response headers, server-side reasoning,
streamed text and tool-call construction. It did not reset when the provider
made progress. A healthy reasoning model could therefore stream continuously
and still be cancelled at 300 seconds.

Using only a resettable idle timeout has the opposite failure mode. Heartbeats
or negligible trickle traffic could retain a shared model-call slot forever.
Tool execution already has a separate parent-controlled lifecycle and should
not be conflated with either model transport limit.

## Decision

- Every normal, child, and compaction model call receives a stream-idle timeout.
  It covers opening the HTTP request through its response headers and each wait
  for the next valid SSE event. The interval restarts after every valid event,
  including reasoning, tool-call, usage, ping, and other protocol events without
  visible text.
- `AgentRunner` keeps a separate hard deadline around the complete provider
  call. It begins after the call acquires the shared model semaphore and never
  restarts.
- The hard deadline includes provider-side queueing, reasoning, streaming,
  authentication refresh, and retry backoff inside that call. Waiting for the
  model semaphore, executing tools, and later model calls are separate scopes.
- The default idle timeout is 300 seconds and the default hard deadline is 3600
  seconds. Both are explicit positive runtime settings.
- A timeout after partial streaming does not persist a partial assistant
  message. Normal calls fail their run; compaction records failure and keeps the
  uncompacted context.

## Consequences

- Slow reasoning and output are allowed while the provider continues emitting
  valid events.
- Silent connections and indefinitely trickling connections both have bounded
  lifetimes.
- Each model/tool loop iteration receives fresh timeout intervals; tool calls
  generated by a model count toward the producing model request, while actual
  tool execution does not.
- Provider adapters must enforce the idle interval at their streaming boundary,
  while the runner remains responsible for the provider-wide hard deadline.

## Alternatives Considered

- **Keep one non-resetting timeout.** Rejected because total response duration
  is not evidence that a streaming reasoning request is stalled.
- **Reset the only timeout after each event.** Rejected because heartbeat or
  trickle traffic could keep a shared model slot forever.
- **Reset only for visible text.** Rejected because reasoning, tool-call
  arguments, usage, and protocol events are also legitimate stream progress.
- **Apply timeout policy to tool execution.** Rejected because tools use the
  separate foreground-to-background and explicit-stop lifecycle from ADR 0010.
